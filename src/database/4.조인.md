# 조인 (JOIN)

> *(SQL 예제는 모두 **MySQL** 기준)*

**조인(JOIN)**: 두 개 이상의 테이블을 연결하는 것.

---

## 조인을 실행하면 물리적으로 무슨 일이 일어날까?

조인은 화면상에서는 "표 두 개를 붙이는 일"처럼 보이지만, 내부에서는 **디스크 → 메모리(RAM) → CPU**로 데이터가 이동하며 비교가 일어나는 물리적 작업이다.

### 1. 데이터는 원래 디스크(HDD/SSD)에 있다

- 테이블의 실제 데이터는 **디스크에 파일로 저장**되어 있다. (MySQL InnoDB 기준 16KB 단위의 **페이지**로 묶여 저장)
- CPU는 디스크의 데이터를 **직접 계산할 수 없다.** 반드시 먼저 **메모리(RAM)로 올려야** 한다.
- 디스크 읽기는 가장 느린 단계라서, 조인 성능은 사실상 **"디스크를 몇 번 읽느냐"** 로 결정된다.

### 2. 필요한 페이지를 RAM(버퍼 풀)으로 올린다

- MySQL은 디스크에서 읽은 페이지를 **버퍼 풀(Buffer Pool)** 이라는 RAM 영역에 캐싱한다.
- 이미 버퍼 풀에 있으면 디스크를 안 읽어도 되므로 **매우 빠르다.** (디스크 I/O 생략)
- 그래서 같은 데이터를 반복 조회하면 두 번째부터 빨라진다.

### 3. CPU가 RAM 위의 행들을 비교한다

- 두 테이블의 행을 꺼내 `E.DEPTNO = D.DEPTNO` 같은 **조인 조건을 CPU가 비교**한다.
- 비교에 쓰는 값은 CPU 캐시·레지스터를 거치며, 조건이 맞는 행만 결과로 모은다.
- 결과가 크거나 `ORDER BY`가 있으면 **임시 테이블/정렬 버퍼**를 RAM에 만들고, RAM이 모자라면 **디스크의 임시 파일로 흘려보낸다(spill).** → 이때 다시 느려진다.

### 저장 계층별 속도 차이 (왜 디스크 읽기를 줄여야 하나)

| 계층 | 역할 | 대략적인 접근 속도 |
| --- | --- | --- |
| CPU 레지스터/캐시 | 실제 비교 연산 | ~1 ns |
| RAM (버퍼 풀) | 데이터 임시 적재 | ~100 ns |
| SSD | 영구 저장 | ~100 µs (RAM의 약 1,000배 느림) |
| HDD | 영구 저장 | ~10 ms (RAM의 약 10만 배 느림) |

> 💡 핵심: **CPU ≫ RAM ≫ 디스크** 순으로 빠르다(디스크가 가장 느리다). 그래서 조인 최적화의 목표는 결국 **디스크에서 읽어 올리는 페이지 수를 줄이는 것**이다.

### 조인이 행을 맞춰보는 방식 (조인 알고리즘)

| 알고리즘 | 동작 | 물리적 부담 |
| --- | --- | --- |
| **Nested Loop Join**<br>(중첩 루프) | 기준 테이블 한 행마다 상대 테이블에서 짝을 찾음 | 인덱스가 있으면 짝을 **바로 찾아** 디스크 읽기 최소화 |
| **Hash Join**<br>(MySQL 8.0.18+) | 한쪽 테이블로 RAM에 **해시 테이블**을 만들고 다른 쪽으로 조회 | 인덱스 없는 등가 조인에 유리. 해시가 RAM보다 크면 디스크로 spill |

> 💡 **인덱스(B+Tree)** 가 있으면 짝이 되는 행이 디스크 어디에 있는지 빠르게 찾아, 테이블 전체를 RAM으로 올리지 않아도 된다. → 인덱스가 조인 속도를 좌우하는 이유.

> ⚠️ FROM 절에 연결 조건 없이 테이블만 나열하면(아래 곱집합), 모든 행 조합을 만들어내느라 **읽는 페이지 수와 CPU 비교 횟수가 폭증**한다.

## FROM 절에 여러 테이블 선언하기

```sql
SELECT *
FROM EMP, DEPT
ORDER BY EMPNO;
```

> ⚠️ FROM 절에 명시한 각 테이블의 행이 **모든 경우의 수로 조합(곱집합)** 되어 출력된다. → 정확도가 떨어진다.
>
> A·B 두 테이블을 조인하려면, 둘을 **정확히 연결해 주는 열**이 하나 필요하다. 연결 조건이 없으면 불필요한 데이터까지 모두 조합되어 출력된다.

## 등가 조인 (INNER JOIN)

테이블을 연결한 후, 각 테이블의 특정 열에서 **값이 일치하는 데이터**를 기준으로 행을 선정하는 방식.
→ **내부 조인(INNER JOIN)** 또는 **단순 조인(SIMPLE JOIN)** 이라고도 부른다.

### 열을 비교하는 조건식으로 조인하기

```sql
SELECT EMP.EMPNO, EMP.ENAME, DEPT.DEPTNO, DEPT.DNAME
FROM EMP, DEPT
WHERE EMP.DEPTNO = DEPT.DEPTNO
ORDER BY EMPNO;
```

### 테이블 별칭(alias) 사용하기

```sql
SELECT e.EMPNO, e.ENAME, d.DEPTNO, d.DNAME
FROM EMP e, DEPT d
WHERE e.DEPTNO = d.DEPTNO
  AND e.DEPTNO = 10
ORDER BY e.EMPNO, d.DEPTNO;
```

### WHERE 절에 추가 조건 사용

```sql
SELECT E.ENAME, D.DNAME, D.LOC
FROM EMP E, DEPT D
WHERE E.DEPTNO = D.DEPTNO
  AND E.SAL >= 2000
ORDER BY E.DEPTNO DESC;
```

## 비등가 조인

`=`이 아닌 **범위 조건(BETWEEN 등)** 으로 연결하는 조인.

```sql
-- 사원의 급여가 급여등급(SALGRADE)의 범위에 속하는지로 연결
SELECT E.EMPNO, E.ENAME, E.SAL, S.GRADE
FROM EMP E, SALGRADE S
WHERE E.SAL BETWEEN S.LOSAL AND S.HISAL;
```

## 자체 조인 (Self Join)

**같은 테이블**을 두 번(별칭을 달리해서) 사용해 조인하는 방식.

```sql
-- 사원과 그 사원의 직속상관(MGR)을 같은 EMP 테이블에서 연결
SELECT E1.EMPNO, E1.ENAME, E1.MGR,
       E2.EMPNO AS MGR_EMPNO,
       E2.ENAME AS MGR_ENAME
FROM EMP E1, EMP E2
WHERE E1.MGR = E2.EMPNO;
```

> ℹ️ `E1.MGR` 값이 `NULL`인 사원(상관이 없는 사람)은 조건에 맞는 짝이 없어 출력되지 않는다.

## 표준 문법으로 배우는 조인

### NATURAL JOIN

두 테이블에서 **이름과 자료형이 같은 열**을 자동으로 찾아 그 열을 기준으로 등가 조인하는 방식.

```sql
-- 일반 등가 조인
SELECT E.EMPNO, E.ENAME, D.DEPTNO, D.DNAME
FROM EMP E, DEPT D
WHERE E.DEPTNO = D.DEPTNO;
```

```sql
-- NATURAL JOIN : DEPTNO를 기준으로 자동 등가 조인
SELECT E.EMPNO, E.ENAME, DEPTNO, D.DNAME
FROM EMP E NATURAL JOIN DEPT D;
```

> 💡 결과는 등가 조인과 같지만, 공통 열(`DEPTNO`)을 자동으로 찾아 연결한다는 점이 다르다. 공통 열은 **별칭 없이** 그대로 쓴다.

### JOIN ~ USING

조인 기준이 되는 **공통 열을 직접 명시**하는 방식.

```sql
SELECT E.EMPNO, E.ENAME, DEPTNO, D.DNAME
FROM EMP E JOIN DEPT D USING (DEPTNO)
WHERE E.SAL >= 3000
ORDER BY DEPTNO, E.EMPNO;
```

> 💡 `USING(DEPTNO)`에 쓴 공통 열은 별칭(`E.`, `D.`) 없이 사용한다.

### JOIN ~ ON

조인 **조건식을 직접 지정**하는 방식. 공통 열도 어느 테이블의 것인지 구체적으로 지칭해야 한다.

```sql
SELECT E.EMPNO, E.ENAME,
       D.DEPTNO, D.DNAME
FROM EMP E JOIN DEPT D ON (E.DEPTNO = D.DEPTNO)
WHERE E.SAL >= 3000
ORDER BY D.DEPTNO, E.EMPNO;
```

| 방식 | 조인 기준 | 공통 열 표기 |
| --- | --- | --- |
| **NATURAL JOIN** | 같은 이름·자료형 열 자동 인식 | 별칭 없이 |
| **USING** | 공통 열 이름 직접 명시 | 별칭 없이 |
| **ON** | 조건식 직접 작성 | 테이블명/별칭으로 지칭 |

## OUTER JOIN (외부 조인)

기준 테이블의 행은, 반대쪽에 짝이 **없어도** 모두 출력하는 조인.

### LEFT OUTER JOIN

**왼쪽** 테이블을 기준으로, 오른쪽 데이터 존재 여부와 상관없이 모두 출력.

```sql
SELECT E1.EMPNO, E1.ENAME, E1.MGR,
       E2.EMPNO AS MGR_EMPNO,
       E2.ENAME AS MGR_ENAME
FROM EMP E1 LEFT OUTER JOIN EMP E2
  ON (E1.MGR = E2.EMPNO)
ORDER BY E1.EMPNO;
```

### RIGHT OUTER JOIN

**오른쪽** 테이블을 기준으로, 왼쪽 데이터와 상관없이 모두 출력.

```sql
SELECT E1.EMPNO, E1.ENAME, E1.MGR,
       E2.EMPNO AS MGR_EMPNO,
       E2.ENAME AS MGR_ENAME
FROM EMP E1 RIGHT OUTER JOIN EMP E2
  ON (E1.MGR = E2.EMPNO)
ORDER BY E1.EMPNO;
```

### FULL OUTER JOIN

LEFT JOIN과 RIGHT JOIN을 결합한 것. 양쪽 모두 짝이 없어도 출력한다.

> ⚠️ **MySQL은 `FULL OUTER JOIN`을 직접 지원하지 않는다.** `LEFT JOIN`과 `RIGHT JOIN` 결과를 `UNION`으로 합쳐 같은 효과를 낸다.

```sql
SELECT E1.EMPNO, E1.ENAME, E1.MGR,
       E2.EMPNO AS MGR_EMPNO,
       E2.ENAME AS MGR_ENAME
FROM EMP E1 LEFT OUTER JOIN EMP E2 ON (E1.MGR = E2.EMPNO)
UNION
SELECT E1.EMPNO, E1.ENAME, E1.MGR,
       E2.EMPNO AS MGR_EMPNO,
       E2.ENAME AS MGR_ENAME
FROM EMP E1 RIGHT OUTER JOIN EMP E2 ON (E1.MGR = E2.EMPNO)
ORDER BY EMPNO;
```

## 연습 문제

<details>
<summary><b>실습)</b> JOIN ~ USING 키워드를 사용할 것. 급여가 3000 이상이며, 직속상관이 반드시 있는 사원을 출력하시오.</summary>

```sql
SELECT E.EMPNO, E.ENAME, DEPTNO, D.DNAME
FROM EMP E JOIN DEPT D USING (DEPTNO)
WHERE E.SAL >= 3000
  AND E.MGR IS NOT NULL
ORDER BY E.EMPNO;
```
